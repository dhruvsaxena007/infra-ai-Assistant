1. Machine Embeddings
Endpoint: Used internally by search, recommendations, RAG, machine creation
Files: embedding_service.py, scripts/generate_marketplace_embeddings.py

How it works
Each machine is converted to rich text via build_machine_search_text() (name, brand, model, category, city, specs — not just description). That text is passed through SentenceTransformer all-MiniLM-L6-v2, producing a 384-dimensional vector stored in MongoDB as embedding.

Concept
Semantic embeddings map text to vectors where similar meaning = closer vectors. This enables “drilling machine” to match “crawler drill” even without exact keyword overlap.

Tech
sentence-transformers (local model, no API call per search)
Cosine similarity via NumPy
Alternatives
Approach	Pros	Cons
OpenAI text-embedding-3-small
Higher quality, multilingual
API cost, latency, dependency
MongoDB Atlas Vector Search
Native DB vector index
Requires Atlas + index setup
BM25 / Elasticsearch
Fast keyword search, no ML
Weak on synonyms and Hinglish
ColBERT / cross-encoder reranker
Better ranking accuracy
Slower, heavier compute
2. Semantic Search
Endpoint: GET /machines/semantic-search?query=...
Files: search_service.py

How it works
Embed the user query.
Compare against all machine embeddings using cosine similarity.
Return top matches sorted by similarity score.
Pure vector search — no category/city/price filters.

Alternatives
FAISS / Annoy for approximate nearest-neighbor at scale (millions of listings).
Hybrid search (BM25 + vectors) — what your intelligent search already does in a richer form.
3. Intelligent Hybrid Search (Main Search Engine)
Endpoint: GET /machines/ai-search?query=...
Files: intelligent_search.py, query_parser.py, category_mapping.py, openai_parser.py

How it works
Pipeline:

User query
  → Extract filters (LLM + rules merged)
  → MongoDB pre-filter (category, city, price, listing type)
  → Semantic search on filtered pool
  → Hybrid score (weighted):
       category 22%, city 18%, brand/model/name 10% each,
       semantic 12%, price 8%, availability 5%, listing 5%, rating 5%
  → Primary match filter (exact category/city only when matches exist)
  → Return top N normalized machines
Filter extraction is dual-layer:

Rule parser (query_parser.py + category_mapping.py) — deterministic, always runs; handles synonyms like JCB→backhoe loader, drill→crawler drill.
OpenAI parser (openai_parser.py) — optional GPT JSON extraction for brand, model, pincode, etc.; merged with rules when available.
Concept
Hybrid retrieval = structured filters (precision) + semantic similarity (recall). Prevents “excavator” results when user asked for “JCB 3DX”.

Alternatives
Approach	When to use
Pure LLM search (send all machines to GPT)
Small catalogs only; breaks at scale
Elasticsearch + learning-to-rank
Production marketplace at 100k+ listings
Re-ranker model (Cohere Rerank, cross-encoder)
Improve top-5 accuracy after retrieval
Fine-tuned domain embedder
When generic MiniLM misses construction terms
4. Category Synonym System
Files: category_mapping.py

How it works
A hand-built synonym map maps user language to canonical lowercase categories:

JCB, 3DX, backhoe → backhoe loader
drill, pneumatic drill, AIR ROC → crawler drill
dumper, tipper → dump truck
roller, compactor → road roller
Also maps real marketplace labels (CRAWLER DRILL) → canonical via marketplace_category_to_canonical().

Detectors extract: city, max_price, brand, model, condition, pincode, listing_type, rent_type, negated categories, override phrases, free/cheaper/list-all intents.

Concept
Domain-specific entity resolution — critical because generic LLMs often misclassify JCB as excavator.

Alternatives
Named Entity Recognition (NER) model trained on construction terms
Knowledge graph (brand → model → category ontology)
LLM with strict JSON schema + validation (what Groq text understanding does partially)
5. Conversational Chatbot
Endpoint: POST /chat
Files: chatbot_service.py, chatbot.py, memory.py

How it works
Multi-turn orchestrator:

Image context injection — if user says “is this available in Jaipur?” after image search, expands query using remembered machine type.
Filter extraction — rules + optional OpenAI.
Session filter merge — follow-ups inherit category/city; new category/city starts fresh search.
Strict category search via search_by_filters() (Mongo + post-filters).
Tiered fallback if no exact match:
Same category, drop budget
Same category, other cities
Same category, no filters
Cross-category semantic alternatives (clearly labeled)
Deterministic reply formatting — real fields: name, category_display, price, availability, brand, condition, year, “rating not available”.
AI Advisor attached to every response.
LangChain memory saves user/assistant turns per session_id.
Search analytics logged to MongoDB search_logs.
Special intents: free request, cheaper (75% budget cut), list all (20 results), override/negation.

Concept
Stateful dialogue + constrained retrieval — memory for context, strict search for accuracy, honest fallback when no exact match.

Alternatives
Approach	Trade-off
Full LLM agent with tool calling
Flexible but can hallucinate machines
RAG over machine catalog
Good for Q&A, weak for structured filters
Redis/Mongo persistent memory
Survives server restarts (your memory is in-RAM today)
LangGraph state machine
More explicit control over multi-step flows
6. Multilingual Text Understanding (Groq LLM)
Endpoints: Used in /chat, /voice/chat, POST /understand-text
Files: text_understanding_service.py, core/groq_client.py

How it works
Sends user message to Groq llama-3.3-70b-versatile with a strict JSON prompt. Extracts:

translated_query, machine_type, brand, model, city, max_price, condition, pincode, listing_type, intent
Supports English, Hindi, Hinglish. Includes hard rules (JCB ≠ excavator).

Used mainly for debug/enrichment in /chat; the chatbot’s actual search relies on the stronger rule parser.

Alternatives
OpenAI GPT-4o-mini with function calling
IndicTrans / NLLB for translation + separate parser
Whisper + rules only (no LLM for text) — cheaper, less flexible
7. OpenAI Intent Parser (Optional)
Files: openai_parser.py

How it works
GPT-4o-mini with response_format: json_object extracts search filters. Merged with rule parser in intelligent search when API key is present. Falls back to rules-only if unavailable.

Alternatives
Groq-only (you already have this elsewhere)
Local small LLM (Phi-3, Llama 3 8B) for offline parsing
8. AI Advisor
Endpoint: Included in /chat response as advisor_message
Files: advisor_service.py

How it works
After search results are confirmed:

Builds structured summaries of top 5 machines (price, availability, condition, year, rating).
Sends to Groq Llama 3.3 with strict rules: don’t invent data, don’t assume rating if null.
Returns short practical recommendation.
Fallback: deterministic “Top pick: …” if Groq fails.
Concept
Grounded generation — LLM only explains pre-retrieved, verified listings.

Alternatives
Template-based advisor (no LLM) — faster, zero hallucination risk
Ranking-only UI (show scores, no prose)
Personalized recommender (collaborative filtering on user history)

10. Image Search AI
Endpoint: POST /image-search
Files: image_search.py, image_classification_service.py

How it works
User uploads image (jpg/png/webp).
MobileNetV2 (ImageNet weights) classifies top-3 labels.
extract_machine_search_intent() maps ImageNet labels → marketplace categories:
drill, drilling → crawler drill
excavator, power shovel → excavator
loader, backhoe → backhoe loader
Broad: construction equipment → multiple earthmoving categories
Unknown → ask user to clarify (no forced wrong category)
Runs intelligent search with detected query.
Saves image context to session for follow-up chat.
Concept
Transfer learning via pre-trained classifier + domain mapping layer. ImageNet wasn’t trained on InfraForge listings, so mapping is heuristic.

Alternatives
Approach	Accuracy for construction
Custom YOLO / EfficientNet trained on your machine photos
Best for production
CLIP / OpenAI Vision
Zero-shot, API cost
Google Vision API labels
Generic, similar to MobileNet
Reverse image search (embedding of image vs listing images)
Match exact listing photos
11. Image Context Memory
Files: image_context_memory.py

How it works
In-memory per-session store of last detected machine type. When user says “is this available in Jaipur?” or “ye wali machine”, replaces “this” with remembered type → crawler drill available in jaipur.

Detects strong references (this, ye, same) vs explicit machine names (doesn’t override if user names excavator).

Alternatives
Full dialog state in Redis with TTL
LLM coreference resolution (“this” → last discussed entity)
12. Recommendation Engine
Endpoint: GET /machines/{id}/recommendations
Files: recommendation_service.py

How it works
Build search text from target machine (build_machine_search_text).
Embed and semantic-search all other machines.
Hybrid recommendation score:
50% semantic similarity
25% same category
15% same city
10% rating (neutral 2.5 if null)
Returns top 5 similar listings.

Alternatives
Collaborative filtering (“users who viewed X also viewed Y”)
Content-based + popularity hybrid (Amazon-style)
Graph recommendations (same seller, same region, same brand family)
13. Price Insight Engine
Endpoint: GET /price-insight/{machine_id}
Files: price_insight_service.py

How it works
Rule-based analytics (not LLM):

Find similar machines (same category, city, listing type via Mongo).
Compare price_per_day (rent) or selling_price (sell) using effective_price().
Compute average, difference, percentage.
Label: Below Market / Fair / Above Market with human-readable recommendation.
Alternatives
ML regression on historical rental prices
Time-series pricing (seasonal demand)
LLM explanation layer on top of stats (your advisor partially does this)
14. Deal Score Engine
Endpoint: GET /deal-score/{machine_id}
Files: deal_score_service.py

How it works
Weighted heuristic score (0–100), not ML-trained:

Base 50
Price vs market average: ±25
Rating if present: ±20; if null: +5 neutral
Availability: +8 / −10
Condition (new/used): +2 to +5
Low security deposit on rent: +3
Labels: Excellent / Good / Average / Poor Deal.

Alternatives
Trained classifier on labeled “good deal” outcomes
Multi-criteria decision analysis (MCDA)
Dynamic pricing model from marketplace transaction data
15. Machine Comparison
Endpoint: GET /compare-machines?machine1_id=&machine2_id=
Files: comparison_service.py

How it works
Rule-based side-by-side:

better_for_budget — lower effective price
better_rating — higher rating (handles null)
overall_recommendation — composite: rating×0.5 + availability×0.2 − price penalty
Returns both full machine objects + verdict fields.

Alternatives
LLM-generated comparison table (grounded on two machine JSONs)
Weighted user-preference comparison (user cares more about price vs year)
16. RAG Document Q&A
Endpoints: POST /rag/upload-pdf, POST /rag/upload-text, POST /rag/ask
Files: rag_service.py, pdf_service.py, rag.py

How it works
Classic Retrieval-Augmented Generation:

Ingest: PDF → pypdf text extraction → split into 500-char chunks.
Index: Each chunk embedded with SentenceTransformer; stored in in-memory list (not MongoDB).
Retrieve: Question embedded → cosine similarity vs all chunks → best chunk (threshold 0.2).
Generate: Best chunk + question sent to Groq Llama 3.3 → answer grounded in context only.
Concept
Answer questions from uploaded manuals, policies, or spec sheets — separate from machine search.

Alternatives

17. Image Quality Check (Blur Detection)
Endpoint: GET /image-quality?image_path=...
Files: image_quality_service.py

How it works
OpenCV Laplacian variance on grayscale image:

Variance < 100 → blurry (poor)
Else → good
Used to warn sellers/users about low-quality listing photos.

Alternatives
BRISQUE / NIQE no-reference quality metrics
Deep learning quality model (KonIQ-10k)
Resolution / aspect ratio checks (simpler rules)
18. Image Hash (Duplicate Detection)
Endpoint: GET /image-hash?image_path=...
Files: image_hash_service.py

How it works
Resize to 8×8 grayscale → MD5 hash of pixel bytes. Exact same resize = same hash; near-duplicates may differ (this is a basic hash, not true perceptual hashing).

Alternatives
pHash / dHash / aHash — proper near-duplicate detection
CLIP image embeddings — semantic similarity between photos
SIFT/ORB feature matching — robust to crop/rotation
19. Conversational Memory (LangChain)
Files: memory.py

How it works
LangChain ConversationBufferMemory per session_id — stores all user/assistant turns in RAM. Used to detect follow-ups and provide history string for future LLM calls.

Clears on server restart.

Alternatives
MongoDB/Redis session store
Sliding window memory (last N turns only)
Summary memory (LLM compresses old turns)

21. Real Data Normalization Layer (AI Support Infrastructure)
Files: machine_mapper.py, machine_normalizer.py, reference_cache.py, machine_repository.py

How it works
Raw InfraForge MongoDB (ObjectId refs, rentalPrice, equipmentCategory) → stable normalized object for all AI services and frontend.

Resolves: category names, brand/model, city, seller, ratings from reviews, images from Cloudinary URLs, rent vs sell pricing.

build_machine_search_text() ensures embeddings work when description is empty.

Without this layer, AI features would crash on null fields or wrong schema.

Summary
Your backend is a hybrid AI marketplace assistant: local embeddings for search scale, Groq LLMs for language understanding and advice, Whisper for voice, MobileNet for image-to-category, and heavy rule-based guardrails so construction-specific terms (JCB, crawler drill, pincode) stay accurate. Price/deal/compare are explainable heuristics, not black-box ML — good for demo and auditability.

The main upgrade paths for production would be: custom vision model for image search, vector DB at scale, persistent session memory, and learning-to-rank once you have enough search click data.
